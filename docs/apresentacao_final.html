Contexto e problema

Duas fontes, três perguntas que ninguém conseguia responder de forma integrada

O Ceará publica dados de execução orçamentária em um banco PostgreSQL e contratos públicos em uma API REST separada. Cruzá-los manualmente é inviável em escala — e sem esse cruzamento, três perguntas de controle ficam sem resposta:

  1. Rastreabilidade financeira — de cada contrato assinado, quanto já foi pago via ordem bancária, e quanto ainda está apenas empenhado? Há contratos pagos acima do valor firmado?
  2. Detecção de anomalias — existem contratos de alto valor via dispensa ou inexigibilidade de licitação? Credores com histórico de infração estão recebendo novos contratos?
  3. Previsão e planejamento — com base no histórico, é possível estimar o volume de pagamentos do próximo trimestre por órgão?

Orquestração

4 DAGs encadeadas por Datasets — sem horário fixo, sem sensor externo

Cada DAG dispara a próxima automaticamente ao publicar seu Dataset de saída (data-aware scheduling do Airflow 2.x). Todas as quatro foram executadas de ponta a ponta com dados reais.

  • dag_bronze_extract — extração incremental do Postgres de origem e da API de contratos, com validação mínima de schema. success
  • dag_silver_transform — consolida todo o histórico incremental da Bronze, deduplica e enriquece. success
  • dag_gold_load — upsert das dimensões (SCD2 em dim_credor) e das fatos no Data Warehouse. success
  • dag_ml_inference — treina/aplica os dois modelos e gera o relatório narrado. success

Engenharia e qualidade

14 testes unitários — e três bugs reais que eles não pegaram (mas a execução real, sim)

A suíte cobre paginação da API, deduplicação Bronze→Silver e montagem das dimensões/fatos com mocks. Rodar contra dados reais revelou problemas que só aparecem em escala:

  • Timezone em datas da API — data_assinatura vinha com offset (-03:00) e quebrava a concatenação com datas naive do Postgres.
  • Fan-out no merge unidade gestora → órgão — 7 unidades trocaram de órgão superior entre anos, duplicando linhas em fato_empenho.
  • Consolidação incremental — a camada Silver lia só o run mais recente da Bronze; corrigido para acumular todo o histórico incremental.

Os três viraram testes de regressão adicionados à suíte.

Limitações conhecidas

Transparência sobre o que ainda não é perfeito

  • Ligação empenho → contrato é best-effort via codcontrato, que frequentemente vem zerado nos dados de origem — não é uma FK íntegra.
  • Previsão trimestral depende da profundidade do histórico carregado; a janela de demonstração favorece o fallback de persistência sobre o XGBoost em boa parte dos órgãos.
  • dim_orgao.esfera é aproximada a partir de nomepoder, já que a origem não tem um campo de esfera explícito nessa tabela.
  • Banco de origem não é containerizado — o pipeline conecta diretamente ao PostgreSQL real fornecido para o curso, em vez de replicar 1,4M linhas localmente.
